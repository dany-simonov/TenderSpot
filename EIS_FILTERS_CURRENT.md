## 8) Цена (жесткий фильтр)

- price < 500000 -> reject (price-below-threshold)

Парсинг цены теперь идет через отдельную очистку (cleanAndParsePrice):

- принимает сырой фрагмент цены из HTML
- удаляет все пробелы, включая неразрывные и HTML-сущности (&nbsp;, &#160;, &#xA0;)
- удаляет валютные обозначения (₽, руб, руб., рублей, российский рубль)
- заменяет запятую на точку
- удаляет все символы кроме цифр и точки
- если точек несколько, оставляет последнюю как десятичный разделитель, остальные убирает
- если результат невалидный, возвращает 0 (без падения приложения)

Дополнительно:

- для извлечения сырой цены используется extractRawPriceFragment:
  - использует Cheerio для парсинга HTML карточки
  - приоритет 1: берет текст из .price-block__value внутри .registry-entry__form
  - приоритет 2: ищет блоки с текстом Начальная/НМЦК и берет следующий блок со значением
  - приоритет 3: ищет любой блок с цифрами и ₽/руб
  - приоритет 4 (ультимативный fallback): берет полный text() карточки и ищет по regex /([\d\s\u00A0&nbsp;]{3,},\d{2})\s*(?:₽|руб|Российский рубль)/i
  - затем fallback по текстовым шаблонам (НМЦК/Цена/руб/₽)
- parseMoney внутри адаптера также использует ту же очистку cleanAndParsePrice

Корневой селектор карточки на этапе разделения страницы:

- сначала .search-registry-entry-block (целиком)
- затем .row.registry-entry__form (если нет контейнера)

Это нужно, чтобы в блок попадали обе колонки (левая с текстом и правая с ценой).

Формирование ссылки на закупку (sourceUrl):

- ссылка больше не генерируется вручную по шаблону
- берется исходный href из тега a внутри карточки (приоритет: notice223/common-info -> regNumber -> первый доступный href)
- если href относительный, он приводится к абсолютному через базовый домен [zakupki.gov.ru](https://zakupki.gov.ru)
- в итоговый объект идет именно извлеченный sourceUrl

Извлечение дедлайна (deadline):

- извлекается полный текст карточки (`fullText`) и применяется regex:
  - `/Окончание\s+подачи\s+заяв(?:ок|ки)[\s\S]*?(\d{2}\.\d{2}\.\d{4})/i`
- если матч найден, дата дедлайна парсится в ISO
- если целевой блок не найден, deadline сохраняется пустым и тендер не отбрасывается по этому признаку
- в логах пишется `deadline not found` для таких карточек

## 9) Дата публикации (жесткий фильтр)

- published вне окна recent (по умолчанию последние 120 дней) -> reject (published-out-of-range)

## 9.1) Дедлайн (жесткий фильтр expired)

- если deadline найден и он строго меньше сегодняшней даты (сравнение только по дате, без времени) -> reject (expired)
- если deadline сегодня или в будущем -> проходит дальше
- если deadline не найден -> проходит дальше

## 9.2) Автоматическая очистка БД от просрочек

- после загрузки ETL вызывается авто-очистка `cleanExpiredTenders()`
- из коллекции удаляются документы с `deadline < today`
- лог: `[auto-clean] Удалено X просроченных тендеров из БД.`

## 11.1) Умный региональный скоринг (жесткий reject)

Для каждого распарсенного тендера считается `relevanceScore`.

Базовые правила:

- score стартует с 0
- если в полном тексте карточки (`fullText.toLowerCase()`) найден маркер из MOSCOW_MARKERS -> `score += 50`
- если первые 2 цифры ИНН заказчика входят в `['77','50','69']` -> `score += 50`
- если ИНН не из `['77','50','69']` и нет позитивного маркера (`MOSCOW_MARKERS` или `POSITIVE_REGION_MARKERS` или целевой индекс) -> `score -= 3000` (ИНН-гильотина)
- если ИНН входит в hard-список нецелевых префиксов (`01-49, 51-68, 70-76, 79, 83, 86, 87, 89, 91-99`) и нет позитивного маркера -> `score -= 5000` (жесткая ИНН-гильотина)
- если в тексте найден маркер из POSITIVE_REGION_MARKERS или индекс Москвы/МО/Твери (101-129, 140-143, 170-172) -> `score += 40`
- если найдены почтовые индексы и есть чужой индекс при отсутствии целевого (10/11/12/14/17) -> `score -= 5000` (гильотина индексов)
- если найден город вне целевой географии через regex `/(?:г|город)[.\s]+([а-яё][а-яё\s-]{2,})/gi` -> `score -= 1000`
- если найден корень из stop-листа через `includes()` в полном тексте карточки -> `score -= 1000`
- если итоговый score < 0 -> жесткий reject с причиной `regional-garbage`

## 18) Two-Stage Pipeline (актуально)

Начиная с текущей версии `browseResultsHumanLike`, отбор работает в два этапа.

Stage 1 (Surface Scan):

- цель: быстрый сбор кандидатов только по карточкам выдачи, без углубления в страницу тендера
- лимит кандидатов: `MAX_STAGE1_CANDIDATES = 400`
- применяются только быстрые фильтры:
  - `price < MIN_PRICE` -> `price-below-threshold`
  - `isExpiredDate(deadline)` -> `expired`
  - `!isRecentDate(published)` -> `published-out-of-range`
  - `containsAny(title + description, NON_B2B_CUSTOMERS)` -> `non-b2b-customer`
  - `containsAny(title + description, HARD_ROOF_EXCLUDE)` -> `hard-roof-excluded`
  - `!externalId || !title` -> `missing-core-fields`
- региональный скоринг, ИНН-гильотина и проверка индексов на Stage 1 не применяются
- кандидаты сортируются по proxy-score:
  - `softHitsInTitle * 10 + min(12, floor(price / 500000))`
- лог Stage 1:
  - `[stage-1] reject ...`
  - `[stage-1] pass: ...`
  - `[stage-1] done: scannedBlocks=..., passed=..., rejected={...}`

Stage 2 (Deep Enrichment):

- цель: углубление по `sourceUrl` с чтением полной страницы тендера
- перед каждым запросом выполняется дополнительная пауза `await humanPause()`, затем `humanFetch(...)`
- лимит итоговой выборки: `MAX_STAGE2_OUTPUT = 100`
- из полной страницы извлекаются:
  - ИНН: `/\bИНН\s*[:]?\s*(\d{10})\b/i`
  - КПП: `/\bКПП\s*[:]?\s*(\d{9})\b/i`
  - индексы: `/\b(\d{6})\b/g` (только из адресного контекста)
  - адресная строка места: `/[Мм]есто\s+(?:нахождения|выполнения|рассмотрения)[^\n]{0,300}/`
- белый список ИНН-префиксов Stage 2:
  - `77, 97, 99, 50, 90, 69`
- жесткие причины reject на Stage 2:
  - `stage2-bad-inn`
  - `stage2-bad-zip`
  - `stage2-residential`
- текстовый региональный блеклист на Stage 2 полностью отключен
- правило INN-гильотины детерминированное:
  - если найден ИНН и его префикс входит в `NON_TARGET_INN_PREFIXES_HARD` (и не входит в белый список Stage 2), карточка отклоняется без текстовых исключений
- правило ZIP-гильотины детерминированное:
  - сначала собирается `addressContext = legalAddress + postalAddress + deliveryPlace`
  - затем ищутся индексы только в `addressContext`
  - далее отбрасывается мусор: `validZips = zips.filter(zip => /^[1-9]\d{5}$/.test(zip))`
  - если в `validZips` есть хотя бы один чужой индекс и нет ни одного целевого (`10*`, `11*`, `12*`, `14*`, `17*`) -> `stage2-bad-zip`
- при успешном прохождении:
  - `inn` обновляется значением из полной страницы (если найден)
  - пересчитывается `relevanceScore`
  - запись попадает в финальный массив
  - лог: `[stage-2] pass: ... | inn=... | score=...`

Извлечение ИНН на Stage 2:

- ИНН ищется по очищенному от HTML-тегов тексту `htmlToText(html)`
- regex применяется к нормализованному plain-text, чтобы теги/разрывы не ломали матчи

Обработка ошибок Stage 2:

- при сетевой ошибке: `[stage-2] fetch-error: ...` и `continue`
- при 3 ошибках `fetch` подряд: аварийный `break` Stage 2 с возвратом уже накопленного результата

Итоговый системный лог:

- `[etl] Stage 2 summary: passed=..., rejected={...}`
- `[etl] Pipeline summary: scannedBlocks={N}, passedStage1={M}, passedStage2={K}. Loaded to DB: {K}.`

Важно:

- раздел 18 отражает приоритетное, актуальное поведение конвейера и используется как источник истины для Two-Stage архитектуры.
- Stage 2 работает по строгой математике (`ИНН + ZIP`) и не использует NLP-поиск регионов в сыром HTML.
